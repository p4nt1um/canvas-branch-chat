# 我在 Obsidian Canvas 上造了一个分叉对话工具，把 AI 聊天变成思维地图

> 你有没有这样的体验：和 AI 聊着聊着，话题就跑偏了？想同时探索几个方向，但线性对话框只能一条路走到黑？

---

## 起点：一个简单的需求

用 AI 辅助思考时，我经常遇到一个痛点：

**一个问题，想从多个角度探索。**

比如问 AI"这个技术方案怎么样"，我想同时知道成本、安全、用户体验三个角度的分析。但在传统的聊天框里，你得：

1. 问完成本，再问安全，再问体验
2. 三条线混在一起，回头找不到哪句是哪个角度的
3. 想在"成本"那条线上继续深挖，但上下文已经被后面的对话冲乱了

**本质问题：线性对话不匹配网状思维。**

人的思考是发散的、分叉的、可以回溯的。但所有 AI 聊天工具都是一根线往下聊。

## 灵感：Flowith 的分叉对话

我发现 Flowith 这个产品做了"分叉对话"——你可以从任意一个对话节点拉出一条新分支，每个分支独立对话，互不干扰。

这个思路太对了。

但我想要的是：**把这个能力搬到我每天都在用的 Obsidian Canvas 上。**

Canvas 本身就是一个无限画布上的节点+连线系统。如果把每个对话变成一个节点，分叉就是拉一条新线——那对话自然就变成了一张可浏览的地图。

## 行动：从 Fork 开始

没有从零写。在 GitHub 上找到了 HinxCorporation/obsidian-canvas-ai——一个已经在 Canvas 上实现 AI 问答的插件。虽然只有单次 Q&A，没有分叉功能，但核心基建（Canvas 节点操作、流式响应、API 对接）都跑通了。

**审计完代码，结论是：可用，改起来不难。**

于是 fork 之，开始改造。

## 探索：边做边想清楚

### 第一版规划：P0 先跑通核心

定了个 MVP 范围——四个必须实现的功能：

1. **从节点分叉**：右键任意节点 → 输入方向 → 在右侧创建 AI 回答分支
2. **方向标注**：连线上显示分支方向（如"从成本角度"）
3. **上下文继承**：新分支自动携带祖先链的完整对话历史
4. **边标签显示**：用 Canvas 原生的 edge.label

### 关键技术决策

**上下文怎么传？**

Canvas 是图结构，一个节点可能有多个父节点。MVP 阶段只走"直系祖先链"——从当前节点沿连线逐级往上追溯，不做跨分支交汇。够用，且不会出错。

**API Key 怎么存？**

最初的设计是明文存在插件配置里。测试时突然意识到：这太不安全了。Obsidian 插件的配置文件就是 data.json，谁都能看。

改成：**设置页填环境变量名，插件从 process.env 读取实际密钥。** 配置文件里只存 DEEPSEEK_API_KEY 这个名字，不存 key 本身。

**模型名称用下拉还是手填？**

最初写了硬编码下拉菜单，但模型名经常变（deepseek-chat、deepseek-coder、还有新出的…），下拉永远是过时的。

改成纯手填。后续版本计划加"连通性测试"按钮——点一下，插件去调 /v1/models 接口，拉回可用模型列表，顺便验证 API Key 对不对。

### 讨论中涌现的新想法

做着做着，新的可能性自然浮现：

**"能不能一次输入多个方向，自动生成多个分支？"**

当然可以。而且更进一步——如果每个分支用不同的 AI 模型回答呢？

这就是"多模型配置体系"的雏形。讨论后迅速展开成一个完整的配置重设计：

- 每个模型独立配置：别名、Provider、Base URL、API Key、颜色、图标、系统提示词、Temperature
- 保存常用组合为"预设组"——一键加载"辩论赛"（分析师 + 魔鬼代言人 + 裁判）
- 分叉时选模型，每个角色用不同颜色标识

**这不是简单的"多模型支持"，而是把插件从"单 AI 问答"升级成"多角色讨论平台"。**

### 三层预设体系

讨论继续深入，发现预设其实分三个层次：

```
Layer 3: 行业场景模板（六顶思考帽 / SWOT / 5Why）
    ↓ 引用多个
Layer 2: 分叉框架预设（成本/安全/体验 三个方向）
    ↓ 每个方向绑定
Layer 1: 模型配置（DeepSeek 分析师 / GPT-4 魔鬼代言人）
```

最强大的形态：选"六顶思考帽"模板 → 自动套用预设的角色组 → 一键生成 6 条分支，每帽一个 AI 角色回答。

## 实现：目前到哪了

### ✅ 已完成

- **P0 核心 MVP**：分叉对话、方向标注、上下文继承、边标签
- **配置优化**：API Key 环境变量方案、模型名称手填
- **三种操作模式**：分叉（横向展开）、追问（纵向延续）、提交（单次问答）

### 🔜 进行中

- 多模型配置体系（P1 #5）
- 角色推断改为元数据驱动（不再靠奇偶位置交替）
- 追问交互优化（在 AI 节点上追问时弹输入框）
- 多方向批量分叉（动态输入框列表，避免长文本换行误判）

### 📋 规划中

- 节点视觉区分（颜色/图标）
- 导出 Markdown
- 分支合并、对话回放
- MCP 工具调用、知识库挂载

## 反思：几个教训

**1. 源码改完必须立即构建**

改了 TypeScript 源码，忘了跑 npm run build。用户问怎么更新插件时才发现 main.js 还是两天前的。构建是流程的一部分，不是事后补的。

**2. 角色推断不能靠奇偶数**

最初为了简单，按节点位置交替判断 user/assistant（第 1 个是 user，第 2 个是 assistant…）。一旦中间插入手动节点或从 AI 节点直接分叉，角色就全乱了。

正确做法：创建节点时写入角色元数据，读取时按元数据判断。类型定义早就有了，只是没用上。

**3. 配置安全要从第一天做起**

API Key 明文存储不是"后面再优化"的事。既然想到了，就立刻改。安全债务积累的利息比技术债更高。

## 写在最后

这个项目的核心理念很简单：

**对话不应该是一条线，而应该是一棵树、一张图、一个可以反复探索的空间。**

Obsidian Canvas 提供了画布，AI 提供了智能，这个插件做的事情就是把两者缝起来——让每次对话都变成一张值得保存、值得回看、值得继续探索的思维地图。

如果你也在用 Obsidian，也在用 AI 辅助思考，欢迎试试。
